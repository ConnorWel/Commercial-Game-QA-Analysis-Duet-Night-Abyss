## 🐞 BUG-DNA-004: [Character/State Machine] "Truffle and Filbert" high-speed dash state persists automatically after exiting a cutscene

**Reporter:** [Connor Wei]
**Date:** 2026-06-24
**Environment/Version:** Duet Night Abyss (Current Live/Beta Build)
**Module:** Character State Machine / Cinematic Interruption / Input Buffer

### 1. Severity & Priority
* **Severity:** Low - Does not break the game permanently and has multiple easy workarounds, but feels unpolished.
* **Priority:** P4

### 2. Pre-conditions
* Truffle and Filbert is in the active party.
* Ultimate ability (Riding/Mount State) is active.
* Player is in proximity to an area or NPC that triggers an interactive cutscene/dialogue.

### 3. Steps to Reproduce
1. Activate Truffle and Filbert's Ultimate ability to enter the "Riding" state.
2. Activate the character's small skill to enter the high-speed movement/dash state.
3. While the high-speed movement is actively ticking, steer the character into the trigger volume for a story cutscene or dialogue NPC.
4. Allow the cutscene to load and play out. 
5. Complete the cutscene and observe the character's behavior upon returning to open-world control.

### 4. Actual Result
Upon exiting the cutscene, Truffle and Filbert remains in the Ultimate Riding state and **automatically continues the high-speed dash without any player input**. 
* Pressing or releasing the `WASD` keys does not cancel the forward momentum.
* **Workaround:** The bug resolves itself automatically if the character collides with an obstacle, the skill gauge fully depletes, or if the player manually inputs another dash/skill command to override the current state.

**Video Evidence:** [https://www.youtube.com/watch?v=a59q2bcSID4&feature=youtu.be]

### 5. Expected Result
When a cutscene is triggered, a global `Input.Clear()` or `State.Interrupt()` function should fire, flushing all active input buffers and movement multipliers. Upon exiting the cutscene, the character should return to a neutral idle state, awaiting fresh player input.

### 6. Investigation Notes & Root Cause Hypothesis
**QA Analysis:**
This is an Input Buffer / State Interruption failure. 
When the cutscene trigger volume intercepts the character, it pauses the game state or forcefully repositions the actor, but fails to send an `OnSkillCancel()` event to the active high-speed movement script. The game caches the "Skill Active = True" boolean. When control is returned to the player, the locomotion script resumes reading from that cached boolean, resulting in autonomous movement until a new physics event (collision) or input event (new key press) overrides it.

## 🐞 BUG-DNA-005: [Animation/Rendering] Severe mesh clipping and IK distortion on ambient NPCs at "Siyan Pavilion" (祀烟阁)

**Reporter:** [Connor Wei]
**Date:** 2026-06-24
**Environment/Version:** Duet Night Abyss (Current Live/Beta Build)
**Module:** Animation / Rendering / Environment

### 1. Severity & Priority
* **Severity:** Medium - Visually jarring and heavily impacts narrative immersion in a main quest area.
* **Priority:** P3 (Medium-High due to main quest visibility)

### 2. Pre-conditions
* Main quest line is active, requiring travel to Siyan Pavilion (祀烟阁).

### 3. Steps to Reproduce
1. Follow the main quest objective to travel to the Siyan Pavilion area.
2. Locate the ambient, non-interactable NPCs populating the area.
3. Observe their idle animations, clothing physics, and limb placements.
4. Attempt to fix the issue by teleporting out of the area and back, or completely relogging into the game.

### 4. Actual Result
Several ambient NPCs in the vicinity suffer from severe visual corruption. 
* Extreme mesh clipping (e.g., body parts clipping through clothing).
* Clothing render failures or physics spazzing.
* Inverse Kinematics (IK) and limb control distortions, resulting in broken bone structures or unnatural joint bending.
* **Note:** This bug is persistent. Teleporting away to unload the chunk and returning, or completely restarting the client, does not resolve the issue.

**Video Evidence:** [youtube.com/watch?v=QIy56GV72_s&feature=youtu.be]

### 5. Expected Result
Ambient NPCs should load with proper vertex weighting, initialized cloth physics, and correct IK constraints, maintaining visual fidelity regardless of player interaction.

### 6. Investigation Notes & Root Cause Hypothesis
**QA Analysis:**
Because relogging and reloading the chunk does not fix the issue, this is likely not a dynamic memory or loading error, but rather a baked asset or LOD (Level of Detail) configuration issue.
* The specific NPC prefabs placed at Siyan Pavilion might be referencing an outdated/corrupted animation controller, or their low-poly LOD meshes have broken skinning weights. 
* Alternatively, if the area uses a specific environmental physics volume (e.g., simulated wind for the pavilion), it might be violently conflicting with the cloth simulation parameters of these specific NPC models.

## 🐞 BUG-DNA-006: [Exploration/Gadget] "Scanning Hub" targeting vector deviates from crosshair when colliding with level geometry

**Reporter:** [Connor Wei]
**Date:** 2026-06-24
**Environment/Version:** Duet Night Abyss (Current Live/Beta Build)
**Module:** Exploration System / Camera & Physics / Raycasting

### 1. Severity & Priority
* **Severity:** Low - Causes minor frustration during exploration, easily resolved by player repositioning.
* **Priority:** P4

### 2. Pre-conditions
* Player has the "Scanning Hub" (扫描中枢) exploration gadget equipped.
* Player is positioned directly adjacent to level geometry (walls, large rocks, or other solid obstacles).

### 3. Steps to Reproduce
1. Stand flush against an obstacle so that the player character/gadget model is visually clipping into or tightly squeezed against the geometry.
2. Aim the camera crosshair at a specific target area.
3. Deploy the Scanning Hub gadget.
4. Observe the actual area/direction being scanned relative to the crosshair placement.

### 4. Actual Result
The scan's physical origin point or directional vector is influenced by the collision with the nearby obstacle. The resulting scan deviates significantly from where the UI crosshair is pointing, causing the scan to miss the intended target area entirely.
* **Workaround:** Moving the character away from all obstacles into open space restores accurate crosshair-to-scan alignment.

**Video Evidence:** [[(https://youtu.be/0O4S7ZU-VWY)](在此处替换为你的YouTube链接)

### 5. Expected Result
The gadget's scan execution should always align perfectly with the screen-center crosshair, regardless of the physical object's clipping state. 

### 6. Investigation Notes & Root Cause Hypothesis
**QA Analysis:**
This is an issue with the Raycast Origin configuration for the gadget.
* Currently, the game appears to be firing the scanning raycast (or spawning the scanning volume) directly from the `Transform.Position` of the physical 3D gadget model in the game world.
* When the player hugs a wall, the gadget model's collision mesh is pushed or offset by the wall's collider. Because the model shifts, the origin point of the scan shifts with it, breaking the alignment with the camera's central axis.
* **Suggested Fix:** The directional raycast for the scan should be decoupled from the 3D model's physical transform. It should be fired from the `MainCamera` forward vector through the screen-center crosshair, treating the 3D gadget model strictly as a visual VFX source rather than the mathematical origin of the exploration logic.
